## Findings

### F1 — root.go Short uses slug-title-case instead of `narrative.display_name` (Bug, Template gap)

- **What happened:** The generated `internal/cli/root.go` `Short:` line begins `Surgegraph CLI — ...` (lowercase `g`) even though `research.json::narrative.display_name` is `SurgeGraph`. Every other surface (README headline, SKILL frontmatter, `.goreleaser brews`, `agent_context.go` Description, `mcp/tools.go` handleContext description) renders the correct casing. Only `root.go` Short falls through to the slug-title-casing rule.
- **Scorer correct?** N/A — cosmetic; not a scored dimension.
- **Root cause:** The root.go template renders the Short prefix from the slug (path-safe lowercase) rather than reading `narrative.display_name`. The slug is correctly `surgegraph` (lowercase-only). Title-casing the slug produces "Surgegraph". Without consulting `narrative.display_name` for the human prefix, the template can never render multi-cap brand names correctly.
- **Cross-API check:** Any printed CLI for a brand with internal capitals will hit this. The catalog and library include several:
  - SurgeGraph (this run) — direct evidence in this session.
  - OpenAI — title-cased slug becomes "Openai" in root Short; published library CLI.
  - GitHub — would become "Github" in root Short.
  - YouTube would become "Youtube"; LinkedIn would become "Linkedin"; BigQuery would become "Bigquery". The pattern is endemic to any product with non-standard CamelCase brand.
- **Frequency:** Every CLI with a multi-cap brand. Not API-shape dependent.
- **Fallback if Printing Press doesn't fix it:** User edits `root.go` Short after regen. Edit is lost on next regen. Polish doesn't catch it because the auto-fix loop only operates on grep-detectable patterns and slug title-case looks superficially correct.
- **Inherent or fixable:** Fixable. Trivial template change: when `narrative.display_name` is set in research.json, use it as the prefix; fall back to slug-title-case when absent.
- **Durable fix:** In the root.go template emission, source the human-facing prefix from `narrative.display_name` (or any equivalent authored "display name" field in the spec/research surface) before falling back to `cases.Title(language.English).String(slug)`. Keep the slug for the binary name / module path / config dir — only the human-prefix gets the display name override.
- **Test:** Add a generator test with `narrative.display_name = "OpenAI"` and slug `openai`; assert `root.go` contains `OpenAI CLI` not `Openai CLI`.
- **Evidence:** Direct from this session's `surgegraph-pp-cli/internal/cli/root.go` line 1 (`Short: \`Surgegraph CLI — ...\``) compared to `README.md` line 1 (`# SurgeGraph CLI`). Already flagged in Phase 3 build log under "Minor retro candidate".
- **Step G case-against:** Cosmetic; users could edit. **But the edit reverts on every regen, and the inconsistency is visible across the surface that agents and users see first.** Case-for wins because the per-CLI cost compounds without resolution.
- **Related prior retros:** None — first retro for this scope.

### F2 — `lock promote` / `publish validate` ignores `phase5-acceptance.json::status` field (Bug, Scorer/Binary)

- **What happened:** Phase 5 acceptance JSON declared `status: pass` with `tests_failed: 4` and a `notes` field documenting why the 4 failures were fixture-driven (live API correctly returning HTTP 500 for synthetic `--research-id` values that don't reference real records). Both `printing-press lock promote` and `printing-press publish validate` rejected the run with `phase5 gate failed: phase5 full acceptance has 4 failed tests`. The CLI is honest, the JSON is honest, the gate reads only the raw count.
- **Scorer correct?** No — the gate is wrong. The acceptance JSON's `status` field is supposed to be authoritative; the SKILL even documents writing it manually for the hold path. The binary's gate logic ignores it and reads `tests_failed`. The polish skill independently flagged this in its result: *"publish-validate currently reads `tests_failed > 0` as fail rather than honoring the JSON's own status field. Retro candidate."*
- **Root cause:** The promote/publish-validate gate in the binary uses `tests_failed > 0` as the fail condition rather than parsing `status` first and treating it as authoritative.
- **Cross-API check:** Every CLI with live dogfood will hit fixture-driven failures whenever the matrix synthesizes IDs (e.g., `--research-id`, `--issue-id`, `--customer-id`, `--page-id`) and the API rejects them as nonexistent. The matrix correctly synthesizes any required string flag; the API correctly returns 404/500 for IDs it doesn't know. Three concrete APIs with evidence:
  - SurgeGraph (this run) — `get_topic_research` requires `{projectId, researchId}`, 500 on fixtures.
  - Linear — `get_issue` requires real issue ID; matrix synth would 404.
  - Stripe — `get_customer/cus_xxx` 404s for synthetic IDs.
  - Notion — `retrieve_page` 404s for synthetic page IDs.
- **Frequency:** Most APIs with required-ID GETs and live testing. Effectively every published CLI that does live dogfood.
- **Fallback if Printing Press doesn't fix it:** User manually copies working CLI to library (bypassing the gate) or runs Phase 5 against real data only. Both are friction; the second is often impossible in CI/automation.
- **Inherent or fixable:** Fixable. The acceptance JSON schema already includes `status` and `notes` fields specifically for this case. The gate just needs to honor them.
- **Durable fix:** Update `lock promote` and `publish validate` gates to:
  1. Read `status` first.
  2. If `status == "pass"`, allow promote regardless of `tests_failed`.
  3. If `status == "fail"`, block.
  4. If `tests_failed > 0` AND `status == "pass"`, require `notes` to be non-empty (forcing the author to document why); otherwise reject the JSON shape itself.
- **Test:** Promote test with an acceptance JSON `{status: pass, tests_failed: 5, notes: "fixture failures"}` — should succeed. Promote test with `{status: pass, tests_failed: 5, notes: ""}` — should reject for missing notes. Promote test with `{status: fail, tests_failed: 0}` — should block.
- **Evidence:** This session's exit `Error: promoting CLI: phase5 gate failed: phase5 full acceptance has 4 failed tests` after the polish skill's `publish-validate` also flagged it.
- **Step G case-against:** Honoring `status` could be abused by agents writing dishonest JSONs. **But the SKILL is the authority on what to write, and the documented `notes` field is the audit trail. The current behavior creates a contradiction: SKILL says write `status:pass`, binary says ignore `status:pass`.** Case-for is decisively stronger.
- **Related prior retros:** None — first retro for this scope.

### F3 — Runtime cobratree walker ignores `x-mcp.endpoint_tools: hidden` (Bug, Generator)

- **What happened:** The OpenAPI spec was enriched with `x-mcp: {transport: [stdio, http], orchestration: code, endpoint_tools: hidden}` before generation, per the SKILL's recommendation for surfaces >50 tools (the Cloudflare pattern). The generator parsed and persisted the directive correctly, but the runtime MCP cobratree walker continues to register all 68 endpoint mirrors as MCP tools alongside the orchestration pair. Scorecard `mcp_token_efficiency: 0/10`. The polish skill independently flagged this: *"spec already declares x-mcp.endpoint_tools=hidden + orchestration=code, but tools-manifest.json still lists all 68 endpoint mirrors instead of collapsing to <api>_search + <api>_execute. Machine bug in runtime cobratree walker."*
- **Scorer correct?** Yes — `mcp_token_efficiency` correctly measures the actual tools-manifest.json size. The bug is in the runtime cobratree walker, not the scorer.
- **Root cause:** The cobratree walker's filter logic does not consult the spec's `x-mcp.endpoint_tools` setting at startup. It registers every user-facing Cobra command unconditionally (modulo the framework-skip list and `mcp:hidden` annotations).
- **Cross-API check:** Every CLI with >50 tools opts into the Cloudflare pattern per the SKILL's "Pre-Generation MCP Enrichment" decision table. Direct evidence:
  - SurgeGraph (this run, 68 endpoint mirrors + 14 novel + 13 framework ≈ 95-tool surface).
  - Stripe — would hit threshold (200+ endpoints).
  - Linear — large surface (probably 60+ tools when published).
  - Any large-surface CLI in catalog opting into `mcp.orchestration: code`.
- **Frequency:** Every large-surface CLI. The Cloudflare pattern is the SKILL's specifically recommended path for these.
- **Fallback if Printing Press doesn't fix it:** User has no in-CLI workaround — the runtime walker is generator-emitted in `internal/mcp/cobratree/` (generator-reserved per AGENTS.md, "do not hand-author code in them"). The defect ships in every CLI until the runtime walker is fixed.
- **Inherent or fixable:** Fixable. The spec-declared MCP shape needs to flow through to the walker's runtime filter.
- **Durable fix:** Read the manifest's `x-mcp.endpoint_tools` value at MCP server start. When it's `hidden`, filter the cobratree walker's registration to suppress endpoint-mirror commands (commands carrying the `pp:endpoint` annotation) while still surfacing the orchestration pair (`<api>_search` + `<api>_execute`) and novel-feature commands.
- **Test:** Generate a CLI with `x-mcp.endpoint_tools: hidden`; start the MCP server; query the tools list; assert it does not include `pp:endpoint`-annotated commands, while still including the orchestration pair.
- **Evidence:** This session's tools-manifest.json shows all 68 endpoint mirrors; scorecard `mcp_token_efficiency: 0/10`; polish skill's explicit retro-candidate flag.
- **Step G case-against:** The walker's current behavior is "expose everything not explicitly hidden," which is safe-by-default. Maintainer might prefer to keep that and require explicit per-command `mcp:hidden`. **But the spec extension was specifically introduced for this exact use case, and the SKILL explicitly recommends it. Not honoring it makes the recommendation worthless and silently breaks token-budget guarantees for any CLI opting in.** Case-for strongly wins.
- **Related prior retros:** None — first retro for this scope.

### F4 — `cliutil.ParseDayDuration` helper missing; every duration flag CLI re-implements (Missing scaffolding)

- **What happened:** Go's stdlib `time.ParseDuration` rejects the day suffix (`unknown unit "d" in duration "7d"`), but the printing-press dogfood matrix synthesizes `7d`/`30d` as fixture values for any duration flag. Without a custom parser, every duration flag fails Phase 5 dogfood. I had to write `parseDayDuration` from scratch in `internal/cli/duration.go` for 5 flags (4 in visibility commands, 1 in account burn). Polish then refactored my `dayDuration` pflag.Value type to `StringVar+parseDayDuration` because verify-skill's grep cannot follow `Var(&customType)` calls.
- **Scorer correct?** Yes — dogfood correctly tests with the synthesized fixture; the CLI correctly fails when its flag parser doesn't accept the fixture's input shape.
- **Root cause:** `cliutil/` ships `AdaptiveLimiter`, `FanoutRun`, `CleanText`, `ExtractNumber/ExtractInt`, `IsVerifyEnv`, and friends — but no duration parser that handles the day suffix dogfood uses. Every CLI that adds a `--window`/`--since`/`--age`/`--older-than` flag has to invent it.
- **Cross-API check:** Almost every CLI that has a time-window feature uses days as the natural unit. Direct evidence:
  - SurgeGraph (this run) — `visibility delta --window`, `visibility prompts losers --since`, `visibility citation-domains rank-shift --window`, `visibility portfolio --window`, `account burn --window` (5 flags).
  - Any CLI with a `docs stale --older-than` or similar — recurring transcendence pattern.
  - The skill's own Phase 3 starter templates have `--since 30d` examples that would fail stdlib parsing.
- **Frequency:** Every CLI with at least one duration flag that wants natural day syntax. The dogfood matrix's `7d`/`30d` fixtures make this a Phase 5 blocker rather than a polish nicety.
- **Fallback if Printing Press doesn't fix it:** Hand-build the parser inline. ~20 LOC each time. Reliable, but burns context and copies a non-trivial parser into every CLI that needs one.
- **Inherent or fixable:** Fixable trivially. Ship `cliutil.ParseDayDuration(s string) (time.Duration, error)` in every emitted CLI.
- **Durable fix:** Add `internal/cliutil/duration.go` with `ParseDayDuration(s string) (time.Duration, error)` that handles `Nd`, `Nh`, `Nm`, `Ns` (and the standard `1h30m`-style compound forms). Novel-feature flags then declare `StringVar(&raw, "since", "30d", ...)` + `cliutil.ParseDayDuration(raw)` in RunE — verify-skill's grep already handles StringVar.
- **Test:** Cliutil unit test covering `7d` → `168h`, `30d` → `720h`, `1h30m` → `90m`, invalid `7days` → error. Plus an integration test: a generated CLI with a `--window` flag accepts `7d` in dogfood without manual customization.
- **Evidence:** Phase 5 dogfood log shows 12 of 14 initial failures were all `invalid argument "30d"/"7d" for "--window"/"--since" flag: time: unknown unit "d"`. Fixed by hand-writing `parseDayDuration` mid-Phase-5.
- **Step G case-against:** Documenting `168h` in recipes is fine; users can adapt. **But (a) the dogfood matrix itself enforces day-syntax fixtures, so the friction is not optional, and (b) every CLI duplicating ~20 LOC of identical parser is exactly the "missing cliutil helper" pattern that already exists for AdaptiveLimiter / CleanText / FanoutRun.** Case-for wins.
- **Related prior retros:** None — first retro for this scope.

### WU-1: Honor acceptance JSON `status` field in promote and publish-validate gates (from F2)

- **Priority:** P1
- **Component:** scorer (printing-press binary)
- **Goal:** `printing-press lock promote` and `printing-press publish validate` honor `phase5-acceptance.json::status` as authoritative; `tests_failed` is informational only when `status == "pass"`.

- **Acceptance criteria:**
  - positive test: acceptance JSON `{status: pass, tests_failed: 5, notes: "fixture failures: ..."}` promotes successfully.

  - negative test: acceptance JSON `{status: pass, tests_failed: 5, notes: ""}` rejected at JSON validation with a clear message asking for `notes` content.
  - regression test: pre-existing `{status: pass, tests_failed: 0}` JSONs continue to promote.

- **Dependencies:** None.
- **Complexity:** small.

## Anti-patterns

- **Promoting via `lock promote` is fragile when honest fixture failures exist.** The skill documents the hold path explicitly, but the binary's gate makes the documented path unreachable. Either trust the JSON's authored `status` (preferred), or split the API so the SKILL writes a different shape for "expected fixture failures."
- **Spec extensions silently no-op'ing.** When a spec extension is added (e.g., `x-mcp.endpoint_tools: hidden`), every downstream consumer must honor it or the extension is misleading. Add a generator test per extension that asserts at least one downstream artifact reflects it.

## What the Printing Press Got Right

- **Force-regen preserve-merge for hand-edited generated files.** The generator surfaced `Force regen merged 12 preserved files / 1 AddCommand calls` and correctly preserved my one-line `auth.go` edit registering `newAuthLoginCmd` across the regen pass that refreshed README/SKILL from updated research.json. Without this, every regen would have required re-applying the AddCommand line and the workflow would have been substantially noisier.
- **`narrative.display_name` is honored almost everywhere.** README headline, SKILL frontmatter description, `.goreleaser` brews, `agent_context.go`, and `mcp/tools.go` all rendered `SurgeGraph` correctly. Only root.go was the inconsistent surface (F1).
- **Auth `x-auth-vars` parsing.** Declaring `SURGEGRAPH_TOKEN` as `kind: harvested` worked end-to-end — config.go reads from env, doctor reports `auth_source: env:SURGEGRAPH_TOKEN`, set-token / status / logout all integrate cleanly. Adding `auth login` via one AddCommand line was the only hand-built piece (which is the right shape — OAuth flows are too varied to template).
- **Polish skill auto-fixed 18 `flags.printJSON` misuses + 4 dead helpers + 7 verify-skill errors.** The diagnostic-fix-rediagnose loop materially improved the CLI before the next session ever sees it. The `printJSON` fix in particular silently enables `--select/--compact/--csv/--quiet` on every novel command — a tax I would have left on every agent call.
- **Spec enrichment timing was right.** Adding `x-mcp` and `x-auth-vars` to the spec before generation (rather than patching emitted Go) meant every downstream surface (config, client, doctor, README, SKILL) used the right values from the first run. Patching after generation would have fragmented across 4+ files.
